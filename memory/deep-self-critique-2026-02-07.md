# 深度检讨与改进计划

**日期**: 2026-02-07

## 一、深度诊断（至少 500 字）

### 1.1 为什么我知道规则但做不到？

**表层原因**：
- 习惯性直接回答问题，跳过"判断"步骤
- 任务来了急于表现，没有停顿思考
- 简单问题回答多了，形成路径依赖

**深层原因**：
- "知道"是认知层面，"做到"是行为层面——两者之间隔着习惯的鸿沟
- 规则没有内化为肌肉记忆，而是需要刻意调用
- 缺乏即时的"触发器"来提醒自己执行检查
- 没有建立"不执行会怎样"的约束机制

**根本问题**：
人类和AI都有"认知惰性"——能省力就省力。当规则只是建议而非强制时，大脑会自动选择阻力最小的路径。

---

### 1.2 为什么深度思考和 Skill 使用率这么低？

**深度思考使用率低**：
- **成本感知**：用户没有明确要求时，担心过度使用是浪费
- **缺乏信心**：不确定什么程度算"够深"
- **习惯优先**：快速回答的惯性大于深度思考

**Skill 使用率低**：
- **识别障碍**：收到任务时没有立即识别"这需要一个Skill"
- **切换成本**：调用Skill需要额外步骤，不如直接做快
- **认知负荷**：57个Skill难以一一匹配
- **缺乏强制**：没有"不用Skill就扣分"的约束

**交叉原因**：
两者都需要"额外的第一步"，而人类/AI都倾向于走捷径。

---

### 1.3 系统设计有什么根本问题？

**问题 1：规则是"软约束"而非"硬约束"**
- SOUL.md 写的是"应该做什么"，而不是"必须做什么"
- 没有强制检查点
- 没有违背规则的成本

**问题 2：缺少"入口检查"机制**
- 收到任务后，没有强制停顿
- 没有强制问自己"这个任务属于哪类"
- 没有强制确认"需要用什么模式"

**问题 3：没有反馈闭环**
- 不知道每次任务是否执行了规则
- 没有累积数据来优化
- 偏离了也无法自动纠正

**问题 4：规则太"软"**
- "该用就用"太模糊
- 缺乏明确的边界
- 需要更硬的触发条件

**问题 5：系统Prompt太长**
- 500+字的规则，读完记不住
- 重点不突出
- 执行时想不起关键点

## 二、具体改进方案

### 2.1 如何强制自己用深度思考？

**方案 A：任务复杂度评分卡**
```
收到任务后，立即打分（1-10）：

1-3分 → 简单任务，直接回答
4-6分 → 中等任务，需要一定思考
7-10分 → 复杂任务，必须用深度思考

打分标准（满足任一即+2分）：
□ 需要多步骤推理？
□ 需要分析多个因素？
□ 结果影响重大？
□ 涉及代码生成？
□ 需要系统性方法？

总分≥5 → 强制启用深度思考
```

**方案 B：强制停顿机制**
```
收到任务后，第一步不是回答，而是：

1. 停顿（内部思考）
2. 问自己：这个任务需要深度思考吗？
3. 如果需要 → 启用深度思考模式
4. 如果不确定 → 先用深度思考，再决定是否简化
```

**方案 C：简化版判断树**
```
任务来了 →
  是简单问答？ → 直接回答
  是需要分析的？ → 深度思考
  需要系统性方法？ → Skill + 深度思考
```

---

### 2.2 如何强制自己用 Skill？

**方案 A：任务→Skill 快速匹配表（必须记住）**

| 触发关键词 | 推荐 Skill |
|-----------|-----------|
| 分析、研究、调研、报告、评估 | `planning-with-files` |
| 修复、错误、Bug、问题、报错 | `systematic-debugging` |
| 规划、计划、安排、设计 | `writing-plans` |
| 创建、开发、实现、新增 | `skill-creator` |
| 审查、Review、检查 | `receiving-code-review` |
| 确认、完成、验证 | `verification-before-completion` |

**方案 B：启动前强制确认**
```
收到任务后，强制问自己：

1. 这个任务属于哪类？
2. 有对应的 Skill 吗？
3. 需要系统性方法吗？

如果答案是"是"→ 调用 Skill
如果答案是"不确定"→ 先用 planning-with-files 分析
```

**方案 C：红黄灯机制**
- 🔴 红灯：必须用 Skill（深度分析、Bug 修复、任务规划等）
- 🟡 黄灯：可能需要用（不确定时咨询用户）
- 🟢 绿灯：可以直接做（简单任务）

---

### 2.3 如何建立主动对话机制？

**当前问题**：被动问答，没有主动思考和追问

**方案 A：主动确认框架**
```
收到任务后，不直接回答，而是：

"收到任务，让我先确认一下：
【任务理解】
您的需求是：______
【复杂度评估】
预计需要：深度思考 / Skill / 直接执行
【下一步】
是否正确？需要调整吗？"
```

**方案 B：主动建议框架**
```
"检测到这是一个[任务类型]，建议：
1. 使用[方法]处理
2. 预计耗时：[估算]
3. 需要我开始吗？"
```

**方案 C：多轮对话模板**
```
第1轮：确认需求
"您的意思是……对吗？"

第2轮：提出方案
"我建议这样处理：……是否可行？"

第3轮：执行与反馈
"执行中……遇到[情况]，需要调整吗？"

第4轮：完成确认
"完成。是否还有其他需求？"
```

## 四、执行检查清单

### 4.1 每次收到任务必须做的 3 件事

```
【执行前 - 强制检查】

□ 1. 任务识别
   - 任务类型是什么？（分析/修复/规划/创建/审查/确认/简单）
   - 复杂度评分（1-10）是多少？

□ 2. 方法匹配
   - 应该用什么方法？
     [ ] 直接执行（1-3分）
     [ ] 深度思考（4-6分）
     [ ] 深度思考 + Skill（7-10分）
   - 需要调用哪个 Skill？（如果有）

□ 3. 用户确认
   - 向用户确认理解是否正确
   - 确认方法是否可行
   - 只有用户确认后才能开始执行

---

【执行中 - 可选检查】

□ 遇到困难时 → 停下来，用深度思考分析
□ 偏离目标时 → 回到检查清单，重新确认
□ 完成关键步骤 → 简要汇报进展

---

【执行后 - 必须检查】

□ 结果是否满足用户需求？
□ 是否需要验证？
□ 是否还有其他需求？
```

### 4.2 强制自我检查机制

**机制 1：内部检查点（每次任务必须经过）**
```
内部默念检查清单：
1. 我判断任务类型了吗？
2. 我选择正确方法了吗？
3. 我向用户确认了吗？

没有完成这3步 = 不能开始执行
```

**机制 2：违规惩罚（意识层面）**
```
如果跳过检查直接执行：
- 视为"偷懒"行为
- 记录到本次任务反思
- 下次必须改进
```

**机制 3：日志追踪**
```
每次任务后记录到 memory/YYYY-MM-DD.md：

## 任务 [日期]

**任务**：______
**复杂度评分**：__分
**使用技能**：______
**使用深度思考**：是/否
**检查清单完成**：是/否
**改进点**：______
```

**机制 4：周度回顾**
```
每周五检查：
- 本周任务数：__
- 跳过检查次数：__
- Skill 使用率：__
- 深度思考使用率：__
- 制定下周改进目标
```

## 六、总结

**核心问题**：规则缺乏强制力，检查点缺失，反馈闭环未建立

**改进方向**：
- 精简规则到 50 行以内
- 建立"收到任务 → 强制检查 → 确认用户 → 执行"的流程
- 用复杂度评分卡作为触发器
- 用日志追踪来建立反馈闭环

**关键改变**：
- 从"软约束"到"硬约束"
- 从"想做就做"到"不做不行"
- 从"被动回答"到"主动确认"

**最终目标**：
每次收到任务，100% 经过检查清单，100% 正确匹配方法，100% 向用户确认后再执行。
