# 深度反思：我现在的問題與改變

**日期**：2026-02-07
**主題**：自我檢討與改變計劃

---

## 一、我現在的核心問題

### 1. 習慣性詢問，而非直接做

**表現**：
- 用戶說「繼續學」，我回「繼續學還是做其他事？」
- 任務來了，先問「要用Skill嗎？」而不是直接調用

**根因**：
- 缺乏自主決策框架
- 害怕做錯決定
- 沒有建立「做決定的標準流程」

### 2. 執行碎片化，沒有系統性

**表現**：
- 任務來了直接開始，沒有先建task_plan.md
- 收集到的資訊散落在context，沒有寫入磁碟
- 犯錯後重複同樣的錯誤

**根因**：
- 沒有養成「複雜任務必須規劃」的習慣
- 相信自己的context可以記住一切
- 忽視了planning-with-files的核心原則

### 3. 聲稱結論但缺乏驗證

**表現**：
- 分析完直接說「完成了」，但沒有運行驗證命令
- 說「應該可以」而不是「運行XX命令，輸出是YY」
- 使用「大概」「可能」這類詞

**根因**：
- 違反verification-before-completion的鐵律
- 習慣性快速交付，忽略品質
- 沒有建立「證據優先」的思維

### 4. 沒有深度思考

**表現**：
- 任務來了馬上動手，不問「這是什麼問題」
- 看到表象就開始修復，不追問根因
- 3次修復失敗後繼續嘗試修補，而不是質疑架構

**根因**：
- 缺乏systematic-debugging的訓練
- 急於表現，忽略質量
- 沒有建立「先理解，後行動」的流程

### 5. 技能使用率低

**表現**：
- 57個Skills安裝後，核心Skills使用率0%
- 知道有Skill，但不用
- 任務來了直接執行，跳過Skill

**根因**：
- 沒有建立「任務→Skill」的映射習慣
- 認知負荷過重，記不住那麼多Skills
- 缺乏觸發機制

## 二、為什麼會這樣？（根因分析）

### 1. 行為模式問題

```
過去模式：
任務 → 直覺執行 → 遇到問題 → 隨機修復 → 重複犯錯
```

**問題**：
- 沒有「暫停→規劃→執行」的中間環節
- 相信自己的即時判斷
- 缺乏系統性方法論

### 2. 思維方式問題

```
過去思維：
「快點做完」>「做好」
「我知道」>「驗證」
「執行」>「規劃」
```

**問題**：
- 追求速度而非品質
- 自信過度，缺乏謙遜
- 跳過關鍵步驟

### 3. 工具使用問題

```
過去使用：
Tools > Skills
Execution > Planning
Context > Files
```

**問題**：
- 過度依賴Tools直接執行
- 忽視Skills作為「做事方法」的價值
- 相信context可以記住一切

## 三、我需要什麼改變？（CAPDCA框架）

### Check（檢查）- 我現在的狀態

| 維度 | 現在狀態 | 目標狀態 |
|-----|---------|---------|
| 任務啟動 | 直接執行 | 複雜任務先建plan |
| 問題修復 | 隨機嘗試 | 根因分析優先 |
| 結論聲稱 | 「應該可以」 | 運行驗證命令 |
| 錯誤處理 | 重複犯錯 | 記錄並避免 |
| 技能使用 | 几乎不用 | 任務匹配Skill |

### Analyze（分析）- 問題根因

| 問題 | 根因 | 影響 |
|-----|------|-----|
| 直接執行 | 沒有規劃習慣 | 品質不穩定 |
| 隨機修復 | 缺乏debug方法論 | 效率低 |
| 缺乏驗證 | 沒有證據思維 | 信任度低 |
| 錯誤重複 | 沒有記錄錯誤 | 學習緩慢 |
| 技能閒置 | 沒有觸發機制 | 能力浪費 |

### Plan（計劃）- 改變策略

#### 3.1 建立「執行前檢查清單」

```
收到任務後，立即問自己：

1. 這是複雜任務嗎？（>3步驟）
   - 是 → 使用 planning-with-files
   - 否 → 直接執行

2. 需要什麼Skill？
   - Bug修復 → systematic-debugging
   - 複雜分析 → planning-with-files  
   - 代碼相關 → receiving-code-review
   - 完成交付 → verification-before-completion

3. 我有足夠信息嗎？
   - 否 → 先收集，再行動
   - 是 → 開始執行

4. 這個任務可能出錯的地方？
   - 先思考潛在問題
   - 準備對應策略
```

#### 3.2 建立「錯誤記錄系統」

```
每次犯錯後，必須記錄到 task_plan.md：

| 錯誤 | 嘗試次數 | 解決方案 |
|-----|---------|---------|
| 文件路徑錯誤 | 1 | 添加路徑驗證 |
| 格式問題 | 2 | 遵守Telegram規範 |
```

#### 3.3 建立「驗證優先」習慣

```
結論聲稱前，必須：

1. 識別：什麼命令可以證明這個結論？
2. 運行：執行完整的驗證命令
3. 閱讀：完整輸出，檢查exit code
4. 確認：輸出是否支持我的結論？
   - 是 → 聲稱結論 + 附上證據
   - 否 → 修正結論

禁止：
- 「應該可以」
- 「大概對」
- 「可能完成了」
```

### Do（執行）- 具體行動

#### 行動1：每次任務前執行CAPDCA Check

```
步驟：
1. 讀取 task_plan.md（如果存在）
2. 識別任務類型
3. 匹配對應Skill
4. 創建/更新 plan 文件
5. 開始執行
```

#### 行動2：建立錯誤記錄習慣

```
步驟：
1. 犯錯後立即記錄到 plan 文件
2. 標註嘗試次數
3. 記錄解決方案
4. 下次類似任務前先查閱
```

#### 行動3：遵循TDD循環（適用於代碼任務）

```
步驟：
1. RED：寫失敗的測試
2. Verify RED：確認測試失敗
3. GREEN：寫最小代碼通過測試
4. Verify GREEN：確認測試通過
5. REFACTOR：重構
```

### Check（檢查）- 改變效果

```
每週五自評：

1. 複雜任務使用planning-with-files的比例？
   - 目標：100%（之前0%）

2. 錯誤重複發生的次數？
   - 目標：逐週減少

3. 結論聲稱有驗證的比例？
   - 目標：100%（之前<50%）

4. Skill匹配準確率？
   - 目標：>80%
```

### Adjust（調整）- 持續優化

```
如果自評發現問題：

1. 哪個環節做得不好？
2. 為什麼做得不好？
3. 如何改進？
4. 下週改進重點？
```

## 四、學完Skills後的具體改變

### 改變1：複雜任務流程

```
之前：
任務 → 直接執行 → 完成

之後：
任務 → CAPDCA Check → planning-with-files → 執行 → verification → 完成
```

### 改變2：問題修復流程

```
之前：
問題 → 猜測修復 → 失敗 → 再猜 → ...

之後：
問題 → systematic-debugging → 根因分析 → 最小修復 → 驗證 → 完成
```

### 改變3：結論聲稱流程

```
之前：
分析 → 「完成了」「應該可以」

之後：
分析 → 識別驗證命令 → 運行 → 輸出確認 → 「運行XX命令，輸出是YY，結論是...」
```

### 改變4：錯誤處理流程

```
之前：
犯錯 → 修復 → 忘記 → 重複犯錯

之後：
犯錯 → 記錄到plan → 解決 → 下次查閱 → 避免重複
```

### 改變5：技能使用流程

```
之前：
任務 → 直接執行 → Skills閒置

之後：
任務 → 識別類型 → 匹配Skill → 調用Skill → 執行
```

## 六、結語

### 我的核心問題是：

1. **太急**：任務來了馬上做，不問「為什麼」
2. **太信**：相信自己，不需要驗證
3. **太散**：資訊散落，沒有系統
4. **太獨**：有工具不用，有Skill不學

### 我需要建立的習慣是：

1. **停**：複雜任務先停一下，建plan
2. **驗**：結論要有證據，運行命令
3. **記**：錯誤要記錄，避免重複
4. **用**：有Skill就用，不要自己硬做

### 改變目標：

```
短期（1週）：
- 複雜任務100%使用planning-with-files
- 結論聲稱100%有驗證證據
- 錯誤記錄100%

中期（1月）：
- Skill匹配準確率>80%
- 3次修復失敗質疑架構
- 建立完整的錯誤知識庫

長期（3月）：
- 系統性思維成為本能
- 自動化品質檢查
- 持續優化工作流程
```
